I reread the updated repo, and my main take is: the structure is still halfway between a source-asset repo and a finished web-asset repo. The six image folders are still sitting at the repo root, while public/images only contains the hero/about images and public/opengraph.jpg; the portfolio still appears to be driven by src/lib/data.ts, which is still using remote Unsplash coverImage and imageUrl values rather than your local portfolio files.  

So for your question:

Should the original screenshot image folders stay in the root directory?

No — not as the long-term final structure.

They can stay there temporarily while you reorganize, but for a clean production repo I would separate images into two systems:

1. Original/source images

These are your untouched screenshots and raw originals.
They should not live in the repo root and not be the files your site renders.

Put them in something like:

source-assets/images/originals/
  beauty/
  commercial-ecom/
  cover/
  editorial/
  homepage/
  red-carpet/

or, if you want them inside the app repo:

assets-source/originals/
  beauty/
  commercial-ecom/
  cover/
  editorial/
  homepage/
  red-carpet/

2. Final web-ready images

These are the cleaned, enhanced, cropped, resized exports that the site actually uses.

Put them in:

public/images/portfolio/
  beauty/
  commercial-ecom/
  cover/
  editorial/
  homepage/
  red-carpet/

That way:
	•	originals stay safe
	•	final exports stay clean
	•	the site only points to final web images
	•	you never mix raw screenshots with publish-ready assets

How to think about it
	•	Original screenshots = archive / working source
	•	Masters = cleaned high-quality working files
	•	Exports = exact website-ready images

If you want to be even more organized, add this inside each category under source-assets:

source-assets/originals/beauty/
source-assets/masters/beauty/
public/images/portfolio/beauty/

That is the cleanest pro workflow.
